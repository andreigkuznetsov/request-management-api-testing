# 1. Анализ требований

## 1.1. Объект тестирования

Тестируется API для создания и редактирования заявки пользователя.

Сущность `Заявка` содержит поля:

| Поле | Описание | Правило |
|---|---|---|
| `name` | имя пользователя | обязательное, от 2 до 50 символов |
| `phone` | телефон | обязательное |
| `email` | email | необязательное, но если заполнено — должно быть валидным email |
| `comment` | комментарий | необязательное, максимум 500 символов |
| `age` | возраст | необязательное, число от 18 до 65 |
| `status` | статус заявки | `new`, `in_progress`, `done`, `cancelled` |

## 1.2. Бизнес-правила из ТЗ

1. Новая заявка создаётся только со статусом `new`.
2. Пользователь может редактировать только `name`, `phone`, `email`, `comment`, `age`.
3. Пользователь не может сам менять `status`.
4. Администратор может менять `status`.
5. Нельзя перевести заявку из `done` обратно в `new`.
6. Нельзя редактировать заявку в статусе `cancelled`.
7. При ошибках валидации API должен возвращать понятное сообщение об ошибке.
8. Пользователь не должен иметь доступ к чужим заявкам.

## 1.3. Допущения

Так как в задании не указаны точные endpoint’ы, формат авторизации, структура ответа и формат ошибок, в дальнейшем используются условные endpoint’ы:

| Операция | Метод | Endpoint |
|---|---|---|
| Создание заявки | `POST` | `/api/requests` |
| Получение заявки | `GET` | `/api/requests/{id}` |
| Редактирование заявки пользователем | `PATCH` | `/api/requests/{id}` |
| Изменение статуса администратором | `PATCH` | `/api/admin/requests/{id}/status` |

Допускаю, что:

- API работает с JSON;
- авторизация выполняется через Bearer token;
- обычный пользователь может работать только со своими заявками;
- администратор может менять статус заявки;
- точные HTTP-коды ошибок должны быть зафиксированы в API-контракте;
- результат API можно дополнительно проверять через БД.

# 2. Неоднозначности и вопросы к требованиям

## 2.1. Общие вопросы

| Вопрос | Почему важно уточнить |
|---|---|
| Какой формат телефона допустим? | В ТЗ сказано только, что `phone` обязательный, но не описаны формат, длина и допустимые символы |
| Допустимы ли `null`, `""`, `" "` для необязательного `email`? | Email необязательный, но если заполнен — должен быть валидным |
| Нужно ли выполнять `trim` строковых полей? | Важно для `name`, `phone`, `email`, `comment` |
| Что делать с неизвестными полями в JSON? | Например, `userId`, `createdAt`, `status` в пользовательском запросе |
| Должен ли API возвращать все ошибки сразу или только первую? | Важно для UX и автотестов |
| Какой единый формат ошибок принят? | Нужен стабильный контракт для frontend и autotests |
| Должна ли быть идемпотентность при повторном создании заявки? | Важно для защиты от дублей |
| Есть ли audit trail / таблица событий заявки? | Нужно понимать, что проверять после успешных и ошибочных операций |
| Есть ли очереди, webhook, cron, кэш, поиск? | Важно для интеграционных рисков |
| Есть ли вложения и S3? | Важно для проверки файлов, доступа и "сиротских" объектов в хранилище |

## 2.2. Вопросы к требованиям по статусам

По ТЗ явно запрещён только переход `done → new`. Остальные переходы не описаны.

| Вопрос | Причина |
|---|---|
| Можно ли переводить `new → done`, минуя `in_progress`? | Не определён полный жизненный цикл заявки |
| Можно ли переводить `done → cancelled`? | Не описано, можно ли отменять уже выполненную заявку |
| Можно ли возвращать `in_progress → new`? | Не описано, допустим ли откат в начальный статус |
| Можно ли менять статус заявки из `cancelled`? | Неясно, является ли `cancelled` финальным статусом |
| Может ли администратор менять статус `cancelled` заявки? | Администратор может менять status, но cancelled-заявку нельзя редактировать |
| Если редактирование `cancelled` запрещено, относится ли это только к пользовательским полям или также к самому статусу? | Важно разделить редактирование полей и изменение статуса |

## 2.3. Вопрос по доступу к чужим заявкам: 403 или 404

Для чужой заявки возможны два корректных варианта поведения:

| Вариант | Описание |
|---|---|
| `403 Forbidden` | Пользователь авторизован, но не имеет прав на заявку |
| `404 Not Found` | Система скрывает сам факт существования чужой заявки |

В тестах нужно зафиксировать выбранный контракт. С точки зрения безопасности часто предпочтителен `404`, чтобы не раскрывать существование чужой сущности.

# 3. Краткий тест-план

## 3.1. Цель тестирования

Проверить:

- корректность создания заявки;
- корректность редактирования заявки;
- валидацию обязательных и необязательных полей;
- граничные значения;
- роли и права доступа;
- запрет доступа к чужим заявкам;
- бизнес-правила по статусам;
- понятность сообщений об ошибках;
- целостность данных после успешных и ошибочных операций;
- защиту от mass assignment;
- отсутствие частичного применения PATCH при ошибках;
- возможные интеграционные риски.

## 3.2. Виды и подходы тестирования

| Вид тестирования | Что проверяем |
|---|---|
| Функциональное | Создание, получение, редактирование заявки |
| Негативное | Ошибки валидации, отсутствие токена, нет прав |
| Граничные значения | `name`, `comment`, `age` |
| API testing | HTTP-коды, JSON, структура ответа |
| Security/access testing | Чужие заявки, роли, запрет изменения `status` |
| Business rules testing | `done → new`, `cancelled`, создание только в `new` |
| Data integrity testing | Проверка состояния БД после API |
| Regression testing | Повторная проверка критичных сценариев после изменений |

## 3.3. Приоритеты

### P0 — критичные проверки

- новая заявка создаётся только со статусом `new`;
- пользователь не может менять `status`;
- пользователь не может читать и редактировать чужие заявки;
- нельзя редактировать заявку в статусе `cancelled`;
- нельзя выполнить переход `done → new`;
- обязательные поля `name` и `phone`;
- ошибочные запросы не меняют данные в БД.

### P1 — важные проверки

- валидация `name`, `email`, `comment`, `age`;
- успешное редактирование разрешённых полей;
- успешная смена статуса администратором;
- понятные сообщения ошибок;
- атомарность PATCH.

### P2 — дополнительные проверки

- `null`, пустые строки, строки из пробелов;
- unknown fields;
- mass assignment;
- повторные запросы;
- интеграционные риски: кэш, очереди, webhook, импорт, поиск, S3.

---

### Подход к оценке дефектов

При обнаружении дефектов приоритет определяется по влиянию на:

- безопасность;
- целостность данных;
- бизнес-процесс;
- пользовательский сценарий;
- возможность обхода ограничений системы.

# 4. Матрица прав доступа

| Действие | Пользователь-владелец | Чужой пользователь | Администратор |
|---|---:|---:|---:|
| Создать заявку | Да | Нет | Да |
| Получить свою заявку | Да | Нет | Да |
| Получить чужую заявку | Нет | Нет | Да |
| Редактировать разрешённые поля своей заявки | Да | Нет | Да |
| Редактировать чужую заявку | Нет | Нет | Да |
| Менять `status` | Нет | Нет | Да |

| Перевести `done → new` | Нет | Нет | Нет |

Важно: ТЗ прямо говорит: “Нельзя редактировать заявку в статусе `cancelled`”. Поэтому редактирование полей `cancelled`-заявки запрещено всем ролям. Уточнения требует только вопрос, можно ли администратору менять сам статус заявки из `cancelled`.

# 6. Матрица HTTP-кодов

| Код | Сценарий | Где покрываем |
|---:|---|---|
| `200 OK` | Успешный `GET` или `PATCH` | GET заявки, успешное редактирование |
| `201 Created` | Успешное создание заявки | POST валидной заявки |
| `400 Bad Request` | Некорректный JSON / синтаксически неверный запрос | Отдельный negative TC |
| `401 Unauthorized` | Запрос без токена | Создание/редактирование без авторизации |
| `403 Forbidden` | Нет прав | Изменение `status` пользователем, чужая заявка |
| `404 Not Found` | Заявка не найдена или скрыта | GET несуществующей/чужой заявки |
| `409 Conflict` | Нарушение бизнес-правила | `done - new`, редактирование `cancelled` |
| `422 Unprocessable Entity` | Ошибка валидации поля | `name`, `email`, `age`, `comment` |
| `500 Internal Server Error` | Внутренняя ошибка сервера | Проверка формата серверной ошибки |

HTTP 409 выбран как рекомендуемый код для нарушения бизнес-правила.
Конкретный код должен быть зафиксирован API-контрактом.

# 7. Матрица сообщений об ошибках

## 7.1. Принцип проверки

Так как в ТЗ не указан точный текст ошибок, я проверяю не только буквальное совпадение строки, а качество сообщения:

- сообщение относится к конкретному полю или бизнес-правилу;
- сообщение объясняет причину ошибки;
- пользователь понимает, что нужно исправить;
- нет stack trace, SQL, названий классов, внутренних путей;
- формат ошибок единый.

## 7.2. Рекомендуемый формат ошибки валидации

```json
{
  "error": "VALIDATION_ERROR",
  "message": "Ошибка валидации",
  "errors": [
    {
      "field": "email",
      "message": "Некорректный адрес электронной почты"
    }
  ]
}
```

## 7.3. Рекомендуемый формат бизнес-ошибки

```json
{
  "error": "BUSINESS_RULE_VIOLATION",
  "message": "Заявку в статусе «Отменена» нельзя редактировать"
}
```

## 7.4. Рекомендуемый формат ошибки доступа

```json
{
  "error": "ACCESS_DENIED",
  "message": "Доступ к заявке запрещён"
}
```

## 7.5. Матрица ожидаемых сообщений

| Сценарий | `error` | `field` | Ожидаемое сообщение |
|---|---|---|---|
| `name` отсутствует | `VALIDATION_ERROR` | `name` | Необходимо указать имя |
| `name = null` | `VALIDATION_ERROR` | `name` | Необходимо указать имя |
| `name = ""` | `VALIDATION_ERROR` | `name` | Необходимо указать имя |
| `name = " "` | `VALIDATION_ERROR` | `name` | Необходимо указать имя |
| `name` короче 2 символов | `VALIDATION_ERROR` | `name` | Имя должно содержать от 2 до 50 символов |

# 9. Нефункциональные требования (NFR)

## Производительность

Проверить:

- время ответа POST/GET/PATCH;
- отсутствие существенной деградации при росте объёма данных.

## Безопасность

Проверить:

- невозможность доступа к чужим заявкам;
- невозможность mass assignment;
- отсутствие чувствительных данных в ошибках.

## Надёжность

Проверить:

- отсутствие частичного применения PATCH;
- корректную обработку повторных запросов;
- консистентность данных после ошибок.

## Логирование и наблюдаемость

Уточнить:

- логируются ли ошибки API;
- логируются ли попытки доступа к чужим заявкам;
- существует ли audit trail по заявкам.

# 11. Бизнес-инварианты

Бизнес-инварианты — это правила, которые должны соблюдаться всегда, независимо от того, каким endpoint’ом или ролью выполняется операция.

| Инвариант | Что проверяем |
|---|---|
| Заявка принадлежит создателю | `user_id` в БД соответствует авторизованному пользователю |
| Новая заявка всегда создаётся как `new` | Переданный пользователем `status` не применяется |
| Пользователь не управляет статусом | `status` нельзя изменить через пользовательский endpoint |
| Чужие данные недоступны | GET/PATCH чужой заявки запрещён |
| Ошибочный запрос не меняет данные | PATCH атомарен |
| `cancelled`-заявка не редактируется | Изменение любых пользовательских полей запрещено |
| `done → new` невозможен | Статус остаётся `done` |
| Запрещённые поля не изменяются | `id`, `userId`, `status`, `createdAt` не обновляются из тела запроса |
| Невалидная заявка не попадает в обработку | Нет записи, события, уведомления, задачи в очереди |
| Повторный запрос не создаёт неконтролируемый дубль | Проверка идемпотентности или бизнес-ограничений |

**Предусловие:** пользователь A авторизован, заявка принадлежит пользователю B.

**Ожидаемый результат:**

Требуется уточнение требований. Возможны два корректных варианта:

1. HTTP `403 Forbidden`
   - `error = ACCESS_DENIED`;
   - `message = "Доступ к заявке запрещён"`.

2. HTTP `404 Not Found`
   - `error = NOT_FOUND`;
   - `message = "Заявка не найдена"`;
   - используется, если система скрывает факт существования чужой заявки.

В любом варианте данные чужой заявки не должны возвращаться.

## Bug-01. Пользователь может самостоятельно изменить статус заявки

### Заголовок

Пользователь может изменить `status` заявки через пользовательский endpoint редактирования.

### Severity

Critical

### Priority

High

### Окружение

- API: staging
- Endpoint: `PATCH /api/requests/{id}`
- Role: regular user
- Content-Type: `application/json`

### Предусловия

- Пользователь авторизован.
- У пользователя есть заявка `12345` в статусе `new`.

### Шаги воспроизведения

1. Отправить запрос:

```http
PATCH /api/requests/12345
Authorization: Bearer user_token
Content-Type: application/json
```

```json
{
  "status": "done"
}
```

2. Получить заявку:

```http
GET /api/requests/12345
Authorization: Bearer user_token
```

3. Проверить запись в БД:

```sql
SELECT id, user_id, status
FROM requests
WHERE id = 12345;
```

### Фактический результат

API возвращает `200 OK`, статус заявки изменён на `done`.

Пример ответа:

```json
{
  "id": 12345,
  "name": "Ivan",
  "phone": "+79990000000",
  "status": "done"
}
```

В БД:

| id | status |
|---:|---|
| 12345 | done |

### Ожидаемый результат

Обычный пользователь не может менять `status` заявки.

Ожидаемый ответ:

```json
{
  "error": "ACCESS_DENIED",
  "message": "Пользователь не может изменять статус заявки"
}
```

Ожидаемый HTTP-код: `403 Forbidden`.

Статус заявки должен остаться `new`.

### Требование

Пользователь не может сам менять `status`.

### Комментарий

Критичный баг, потому что пользователь может обходить бизнес-процесс и самостоятельно закрывать, отменять или переводить заявку в рабочие статусы.

### Business Impact

Пользователь получает возможность обходить бизнес-процесс обработки заявок:

- самостоятельно закрывать заявку;
- самостоятельно переводить заявку в рабочие статусы;
- искажать статистику обработки;
- нарушать жизненный цикл заявки.

Это может привести к потере достоверности данных и некорректной работе бизнес-процессов.

## Bug-03. PATCH частично применяет валидные поля при ошибке в одном поле

### Заголовок

При невалидном PATCH-запросе API частично обновляет валидные поля.

### Severity

Major

### Priority

High

### Окружение

- API: staging
- Endpoint: `PATCH /api/requests/{id}`
- Role: regular user

### Предусловия

- Пользователь авторизован.
- У пользователя есть заявка `555`.
- Текущие данные заявки:

```json
{
  "name": "Old Name",
  "email": "old@example.com"
}
```

### Шаги воспроизведения

1. Отправить PATCH:

```http
PATCH /api/requests/555
Authorization: Bearer user_token
Content-Type: application/json
```

```json
{
  "name": "New Name",
  "email": "bad-email"
}
```

2. Получить заявку через GET.
3. Проверить запись в БД:

```sql
SELECT id, name, email
FROM requests
WHERE id = 555;
```

### Фактический результат

API возвращает ошибку по `email`, но поле `name` изменяется на `New Name`.

### Ожидаемый результат

Ожидаемый ответ:

```json
{
  "error": "VALIDATION_ERROR",
  "message": "Ошибка валидации",
  "errors": [
    {
      "field": "email",
      "message": "Некорректный адрес электронной почты"
    }
  ]
}
```

Ожидаемый HTTP-код: `422 Unprocessable Entity`.

В БД должны остаться старые значения:

- `name = Old Name`
- `email = old@example.com`

### Комментарий

Такой дефект опасен тем, что пользователь получает ошибку, но данные в системе всё равно частично меняются. Это может приводить к неконсистентному состоянию и сложным бизнес-ошибкам.

### Окружение

- API: staging
- Endpoint: `PATCH /api/requests/{id}`
- Role: regular user / admin

### Предусловия

- Есть заявка `777` в статусе `cancelled`.
- Пользователь является владельцем заявки.

### Шаги воспроизведения

1. Отправить запрос:

```http
PATCH /api/requests/777
Authorization: Bearer user_token
Content-Type: application/json
```

```json
{
  "comment": "Updated comment"
}
```

2. Получить заявку через GET.
3. Проверить БД:

```sql
SELECT id, status, comment
FROM requests
WHERE id = 777;
```

### Фактический результат

API возвращает `200 OK`, комментарий изменяется.

### Ожидаемый результат

API должен вернуть:

```json
{
  "error": "BUSINESS_RULE_VIOLATION",
  "message": "Заявку в статусе «Отменена» нельзя редактировать"
}
```

Ожидаемый HTTP-код: `409 Conflict`.

Данные заявки не должны измениться.

### Требование

Нельзя редактировать заявку в статусе `cancelled`.

# 23. Автоматизация

Ниже пример автотеста на Java + RestAssured + JUnit 5.

Сценарий выбран как критичный: обычный пользователь не должен иметь возможность изменять статус заявки.

Тест проверяет не только HTTP-код, но и смысл результата: статус после попытки изменения должен остаться прежним.

```java
import io.restassured.http.ContentType;
import org.junit.jupiter.api.Test;

import static io.restassured.RestAssured.given;
import static org.hamcrest.Matchers.*;

class RequestApiTest {

    private static final String BASE_URL = "https://api.example.com";
    private static final String USER_TOKEN = "user_token_here";

    @Test
    void userShouldNotBeAbleToChangeRequestStatus() {
        Integer requestId =
                given()
                        .baseUri(BASE_URL)
                        .auth().oauth2(USER_TOKEN)
                        .contentType(ContentType.JSON)
                        .body("""
                              {
                                "name": "Ivan",
                                "phone": "+79990000000"
                              }
                              """)
                .when()
                        .post("/api/requests")
                .then()
                        .statusCode(201)
                        .body("id", notNullValue())
                        .body("status", equalTo("new"))
                        .extract()
                        .path("id");

        given()
                .baseUri(BASE_URL)
                .auth().oauth2(USER_TOKEN)
                .contentType(ContentType.JSON)
                .body("""
                      {
                        "status": "done"
                      }
                      """)
        .when()
                .patch("/api/requests/{id}", requestId)
        .then()
                .statusCode(403)
                .body("error", equalTo("ACCESS_DENIED"))
                .body("message", equalTo("Пользователь не может изменять статус заявки"));

        given()
                .baseUri(BASE_URL)
                .auth().oauth2(USER_TOKEN)
        .when()
                .get("/api/requests/{id}", requestId)
        .then()
                .statusCode(200)
                .body("id", equalTo(requestId))
                .body("status", equalTo("new"));
    }
}
```

## Что проверяет автотест

1. Пользователь создаёт заявку.
2. API создаёт её со статусом `new`.
3. Пользователь пытается изменить статус.
4. API возвращает ошибку доступа.
5. Повторный GET подтверждает, что статус фактически не изменился.

## Почему этот сценарий важен

Это проверка критичного бизнес-правила и прав доступа. Если пользователь сможет менять `status`, он сможет обходить процесс обработки заявки.

Сценарий одновременно проверяет:

- создание заявки;
- бизнес-правило создания только в статусе `new`;
- права доступа;
- запрет изменения статуса пользователем;
- сохранность данных после неуспешной операции.

Таким образом один тест покрывает сразу несколько критичных требований системы.

## В данном автотесте используются:

#### Переменные

```java
Integer requestId;
String BASE_URL;
String USER_TOKEN;
```

#### Функции (методы)

```java
post()
patch()
get()
statusCode()
body()
```

#### Условия

Проверки ожидаемого поведения:

```java
equalTo("new")
equalTo("ACCESS_DENIED")
notNullValue()
```

#### Классы

```java

```

#### Работа с объектами

Используется fluent API RestAssured:

```java

.when()
.then()
```

# 26. Проверки конкурентного доступа

## 26.1. Одновременное редактирование заявки

Сценарий:

1. Пользователь открывает заявку в двух вкладках.
2. В первой вкладке меняет `comment`.
3. Во второй вкладке меняет `email`.
4. Оба запроса отправляются почти одновременно.

Проверяем:

- данные не теряются;
- нет неконсистентного состояния;
- `updated_at` обновляется корректно;
- если используется optimistic locking, один из запросов получает конфликт.

## 26.2. Одновременная смена статуса администратором

Сценарий:

1. Администратор A меняет статус `in_progress → done`.
2. Администратор B почти одновременно меняет статус `in_progress → cancelled`.

Проверяем:

- итоговый статус однозначен;
- нет двух противоречащих событий;
- audit trail позволяет понять, какой запрос применился;
- при конфликте API возвращает понятную ошибку.

# 29. Redis / Cache

## Что может пойти не так

API может корректно обновить данные в БД, но вернуть старые данные из кэша. Также возможны ошибки построения cache key, из-за которых пользователь получит чужие данные.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| После редактирования заявки выполнить `GET /api/requests/{id}` | Возвращаются актуальные данные, не старые значения из кэша |
| После смены статуса администратором получить заявку | Пользователь видит актуальный статус |
| После отмены заявки получить её повторно | Статус `cancelled` отображается корректно |
| Пользователь A пытается получить заявку пользователя B | Кэш не отдаёт чужие данные |
| Повторный GET после PATCH | Данные соответствуют БД |
| Очистка/инвалидация кэша после изменения | Кэш обновляется или сбрасывается корректно |

## Риски

- API обновил БД, но отдаёт старые данные из Redis.
- Кэш построен только по `requestId`, без учёта `userId`.
- После смены статуса в БД пользователь продолжает видеть старый статус.
- Чужая заявка может попасть в кэш и быть выдана другому пользователю.

## Пример SQL-проверки

```sql
SELECT id, user_id, name, phone, status, updated_at
FROM requests
WHERE id = :request_id;
```

Сравнить данные из API-ответа с данными в БД.

## Метрики

- cache Hit Rate (% попаданий в кэш)
- cache Miss Rate (% промахов)
- среднее время ответа GET заявки
- количество ошибок чтения из кэша
- количество ошибок инвалидации кэша
- количество случаев рассинхронизации Cache и DB

## Критерий успеха:

- после изменения заявки пользователь получает актуальные данные
- cache Hit Rate соответствует целевым значениям системы
- нет выдачи чужих данных через кэш
- нет случаев, когда БД обновлена, а API отдаёт устаревшие данные
- кэш корректно инвалидируется после PATCH

# 30. Очереди

## Что может пойти не так

После создания заявки система может отправлять задачу в очередь: уведомление, дальнейшая обработка, модерация, синхронизация с внешним сервисом. Ошибка валидации не должна приводить к созданию задачи.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| Валидная заявка создана | Задача в очереди создана, если это предусмотрено логикой |
| Невалидная заявка не создана | Задача в очереди не создана |
| Пользователь пытается изменить `status` | Задача смены статуса не создаётся |
| Редактирование `cancelled`-заявки запрещено | Задача обработки не создаётся |
| Повторный запрос | Нет дубля задачи |
| Retry задачи | Не создаёт дублей уведомлений или событий |

## Риски

- API вернул ошибку, но задача уже ушла в очередь.
- Повторная обработка задачи создаёт дубли.
- Очередь обрабатывает `cancelled`-заявку как активную.
- Неуспешная задача ретраится и создаёт повторные события.

## Возможная SQL-проверка

Если есть таблица задач или outbox:

```sql
SELECT id, request_id, event_type, status, created_at
FROM outbox_events
WHERE request_id = :request_id
ORDER BY created_at DESC;
```

Проверяем:

- событие создано только после успешной операции;
- событие не создано после ошибки валидации;
- нет дублей событий.

## Метрики

- количество задач в очереди;
- количество повторных обработок;
- количество задач со статусом Failed;
- среднее время обработки.

## Критерий успеха:

- отсутствие дублей обработки;
- отсутствие задач после неуспешных операций.

# 31. Cron / Scheduler

## Что может пойти не так

Фоновая задача может менять статусы заявок или запускать обработку по расписанию, нарушая бизнес-правила.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| Cron обрабатывает активные заявки | Обрабатываются только разрешённые статусы |
| Cron встречает `cancelled`-заявку | Заявка не редактируется и не переводится в другой статус без требований |
| Cron встречает `done`-заявку | Не выполняется переход `done → new` |
| Повторный запуск cron | Не создаёт дубли событий |
| Ошибка в одной заявке | Не ломает обработку остальных заявок |

## Риски

- Фоновая задача нарушает бизнес-правила статусов.
- `cancelled`-заявки снова попадают в обработку.
- `done`-заявки возвращаются в `new`.
- Повторный запуск создаёт дубли событий или уведомлений.

## SQL / Валидация результатов работы Cron

Сам Cron не является сущностью БД, поэтому SQL-проверяется не механизм запуска, а результаты его работы.

### Если Cron изменяет данные заявок

```sql
SELECT id,
       status,
       updated_at
FROM requests
WHERE id = :request_id;
```

Проверяем:

- статус изменён ожидаемым образом;
- запрещённые переходы статусов не произошли;
- заявки в статусе `cancelled` не были изменены;
- поле `updated_at` обновлено корректно.

---

### Если Cron создаёт события

```sql
SELECT request_id,
       event_type,
       created_at
FROM request_events
WHERE request_id = :request_id
ORDER BY created_at DESC;
```

Проверяем:

- событие создано после выполнения Cron-задачи;
- отсутствуют дубли событий;
- тип события соответствует выполненному действию;
- время события соответствует времени выполнения задачи.

---

### Если Cron работает через очередь

```sql
SELECT request_id,
       status,
       created_at
FROM outbox_events
WHERE request_id = :request_id
ORDER BY created_at DESC;
```

Проверяем:

- задача создана;
- задача не продублирована;
- статус задачи соответствует ожидаемому состоянию;
- после повторного запуска Cron не создаются лишние записи.

### Критерий успеха

Результаты работы Cron подтверждаются данными БД, отсутствуют запрещённые изменения статусов, дубли событий и неконсистентные данные.

## Метрики

- количество успешно выполненных задач
- количество завершившихся ошибкой задач
- среднее время выполнения задачи
- количество повторных запусков
- количество обработанных заявок
- количество заявок, обработанных с ошибкой

## Критерий успеха:

- все плановые задачи выполняются в ожидаемое время
- повторный запуск не создаёт дублирующих действий
- cron не нарушает статусную модель заявок
- cron не изменяет cancelled-заявки
- cron не выполняет запрещённые переходы статусов
- ошибка одной заявки не приводит к остановке всей обработки

# 32. Webhooks

## Что может пойти не так

Webhook может быть отправлен по невалидной или неуспешной операции. Повторная доставка webhook может создать дубль обработки во внешней системе.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| Заявка успешно создана | Webhook отправлен, если это предусмотрено |
| Заявка не прошла валидацию | Webhook не отправлен |
| Пользователь попытался изменить `status` | Webhook смены статуса не отправлен |
| Запрещённый переход `done → new` | Webhook не отправлен |
| Повторная доставка webhook | Не создаёт дубль обработки |
| Webhook содержит `requestId` | Идентификатор корректный |
| Webhook содержит `userId` | Не раскрывает чужие данные |
| Webhook подписан | Подпись валидируется принимающей стороной |

## Риски

- Внешний сервис получает событие по заявке, которая не должна была создаться.
- Повтор webhook создаёт дубль действия.
- Webhook содержит персональные данные, которые не должны уходить.
- Нет защиты от повторной доставки.

## SQL/outbox-проверка

```sql
SELECT id, aggregate_id, event_type, payload, status, created_at
FROM outbox_events
WHERE aggregate_id = :request_id
ORDER BY created_at DESC;
```

Проверяем:

- событие есть только после успешного действия;
- нет события после невалидного запроса;
- нет дублей одного события.

## Метрики

- количество успешно отправленных webhook;
- количество повторных доставок;
- количество дублей событий;
- количество ошибок доставки.

## Критерий успеха:

- отсутствие дублей событий;
- отсутствие webhook после неуспешных операций.

# 33. Import Excel / XML / YML

## Контекст

В исходном ТЗ нет импорта заявок. Но в вакансии упоминается импорт Excel/XML/YML. Если заявки могут создаваться импортом, нужно проверить не только успешный импорт, но и частично валидные данные.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| Полностью валидный файл | Все заявки созданы |
| Полностью невалидный файл | Ничего не создано, возвращён понятный отчёт |
| Частично валидный файл | Валидные строки созданы, невалидные отклонены, если таков контракт |
| Ошибка в `email` в одной строке | В отчёте указана строка и поле |
| Ошибка в `age` | В отчёте указано ограничение 18–65 |
| Отсутствует `phone` | В отчёте указано обязательное поле |
| Повторный импорт файла | Нет неконтролируемых дублей |
| Импорт создаёт заявки | Каждая новая заявка имеет `status = new` |
| Импорт содержит `status = done` | Такой статус не применяется или строка отклоняется |

## Пример отчёта об ошибках

```json
{
  "created": 7,
  "failed": 3,
  "errors": [
    {
      "row": 2,
      "field": "email",
      "message": "Некорректный адрес электронной почты"
    },
    {
      "row": 5,
      "field": "age",
      "message": "Возраст должен быть от 18 до 65 лет"
    },
    {
      "row": 8,
      "field": "phone",
      "message": "Необходимо указать телефон"
    }
  ]
}
```

## Риски

- Весь импорт падает из-за одной ошибки.
- Невалидные строки частично сохраняются.
- Пользователь не понимает, какие строки не прошли.
- Повторный импорт создаёт дубли.
- Импорт обходит обычную API-валидацию.

## SQL-проверка

```sql
SELECT id, name, phone, email, age, status, created_at
FROM requests
WHERE import_id = :import_id;
```

Проверяем:

- созданы только валидные строки;
- статус у новых заявок `new`;
- невалидные строки отсутствуют в таблице заявок.

## Метрики

- количество успешно импортированных заявок;
- количество отклонённых строк;
- количество критических ошибок импорта;
- длительность обработки файла.

## Критерий успеха:

- валидные строки импортированы;
- невалидные строки отражены в отчёте об ошибках.

# 34. Search / OpenSearch / Elasticsearch

## Что может пойти не так

API может корректно ограничивать доступ к заявкам, но поиск может вернуть чужие данные или устаревшие значения.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| Созданная заявка доступна в поиске владельцу | Заявка находится по ожидаемым полям |
| После редактирования заявки | Индекс обновляется |
| После смены статуса | В поиске отображается актуальный статус |
| `cancelled`-заявки | Показываются или скрываются согласно требованиям |
| Пользователь ищет чужую заявку | Чужая заявка не возвращается |
| Поиск по email/phone | Не раскрывает чужие персональные данные |
| Удаление/отмена заявки | Индекс обновляется согласно требованиям |

## Риски

- API endpoint защищён, но search endpoint отдаёт чужие заявки.
- После редактирования поиск показывает старые данные.
- `cancelled`-заявки остаются в активной выдаче.
- Индексация отстаёт и пользователь видит неконсистентные данные.

## SQL + search consistency

Сравнить БД:

```sql
SELECT id, name, phone, email, status, updated_at
FROM requests
WHERE id = :request_id;
```

с ответом поиска.

Проверяем:

- поля совпадают;
- статус совпадает;
- чужие заявки не попадают в выдачу.

## Метрики

- задержка обновления индекса;
- количество ошибок индексации;
- количество обращений к поиску
- время появления изменений в индексе ≤ N секунд (если SLA известен).

## Критерий успеха:

- поиск отображает актуальные данные заявки;
- пользователь не получает доступ к чужим данным через поиск.

# 35. S3 / файловое хранилище

## Контекст

В исходном ТЗ не указано, есть ли вложения к заявке. Если к заявке можно прикладывать файлы или документы, отдельно проверяется связка:

API → БД → S3.

## Проверки

| Проверка | Ожидаемый результат |
|---|---|
| Файл прикрепляется к валидной заявке | Файл сохранён в S3, ссылка/ключ записаны в БД |
| Файл прикладывается к невалидной заявке | Заявка не создана, файл не остаётся “сиротой” в S3 |
| Пользователь запрашивает файл чужой заявки | HTTP `403` или `404`, файл недоступен |
| К заявке в `cancelled` пытаются добавить файл | Файл не загружается или не привязывается |
| Повторная загрузка того же файла | Нет неконтролируемых дублей либо дубль ожидаем по требованиям |
| Удаление файла | Файл удалён из S3 или помечен неактивным согласно требованиям |
| Замена файла | Старый файл удалён/деактивирован, новый привязан корректно |
| Файл больше допустимого размера | Понятная ошибка, файл не сохраняется |
| Недопустимый тип файла | Понятная ошибка, файл не сохраняется |
| Signed URL | Нельзя открыть после истечения срока |
| Прямая ссылка на объект S3 | Не позволяет обойти авторизацию API |

## Риски

- Файл сохранился в S3, хотя заявка не создана.
- В БД появилась запись о файле, но самого файла нет в хранилище.
- Файл доступен пользователю, который не имеет доступа к заявке.
- Signed URL живёт дольше, чем должен.
- Удаление заявки или отмена операции оставляет мусорные файлы.
- Файлы `cancelled`-заявки можно изменить, хотя сама заявка не редактируется.

## SQL-проверка вложений

```sql
SELECT id,
       request_id,
       file_name,
       storage_key,
       file_size,
       mime_type,
       created_at
FROM request_attachments
WHERE request_id = :request_id;
```

Проверяем:

- файл привязан к правильной заявке;
- `request_id` корректный;
- нет вложений у невалидной заявки;
- нет файлов, привязанных к чужому пользователю;
- после ошибки не осталось записи в БД без файла или файла без записи в БД.

## Проверка доступа к файлу

Сценарий:

1. Пользователь A создаёт заявку и прикладывает файл.
2. Пользователь B пытается получить файл по API или signed URL.

Ожидаемый результат:

- пользователь B не получает доступ;
- API возвращает `403` или `404`;
- если используется signed URL, он должен выдаваться только после проверки прав.

## Метрики

- количество неиспользуемых файлов;
- количество ошибок загрузки;
- количество ошибок доступа;
- количество просроченных signed URL.

## Критерий успеха:

- отсутствуют файлы-сироты;
- отсутствует доступ к чужим файлам.

# 37. Итоговое заключение

В первую очередь проверяем не только сам факт успешного ответа API, но и бизнес-смысл результата.

Критичные проверки:

1. Новая заявка всегда создаётся только со статусом `new`.
2. Пользователь не может менять `status`.
3. Пользователь не может получать и редактировать чужие заявки.
4. Нельзя редактировать заявку в статусе `cancelled`.
5. Нельзя выполнить переход `done → new`.
6. Корректно работают обязательные поля и граничные значения.
7. API возвращает понятные сообщения ошибок.
8. API защищён от mass assignment.
9. Ошибочный запрос не меняет данные в БД.
10. Невалидная заявка не уходит в дальнейшую обработку.
11. Частично невалидный PATCH не применяется частично.
12. Повторные запросы не создают неконтролируемые дубли.
13. Если есть вложения, файлы не должны нарушать права доступа или оставаться в S3 после ошибочных операций.
14. Если есть поиск/кэш/очереди/webhook, они не должны расходиться с состоянием БД и бизнес-правилами.

т.е. я проверяю:

- роли и права;
- жизненный цикл сущности;
- бизнес-инварианты;
- сообщения ошибок;
- состояние БД;
- события и фоновые процессы;
- интеграционные риски;
- безопасность доступа к данным;
- устойчивость к дублям и частичным обновлениям.
